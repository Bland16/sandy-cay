# Sandy Cay — Blind Use Cases, Pass 3 (B41–B80)

**Author:** independent QA analyst. Written from `SPEC.md` and `FRONTEND-SPEC.md` only — no implementation was read.
**Scope note:** deliberately avoids the everyday-scenario pass and B1–B40. This pass aims at the maths, the degenerate configurations, and the seams between features. `UNSPECIFIED` marks places where the spec genuinely does not decide; those are flagged as questions, not defects.
**Convention:** week under test is Mon 2026-07-13 → Sun 2026-07-19 unless stated. Config is the §8 default unless stated.

## A. The scheduling maths itself (B41–B48)

### B41 — All four weights tie at zero after renormalization
| | |
|---|---|
| **Setup** | Cabana → Tuning. Set `weights.proximity = 0`, `balance = 0`, `stability = 0`, `preference = 0`. One flexible task "Read Chapter 4" (60 min), Tue is empty 08:00–18:00. |
| **User action** | Week `⋯` → Re-optimize. |
| **Expected** | No divide-by-zero, no NaN, no crash. Every candidate slot scores identically (0, or an equal defined value), so §2.3's tiebreak applies verbatim: **"ties → earlier"**. "Read Chapter 4" lands at the earliest legal slot of the earliest legal day — Mon 08:00. Placement is stable across repeat Re-optimizes. |
| **Why it's risky** | §2.3 says "Weights renormalized" — the obvious implementation is `w_i / Σw`, and `Σw = 0` yields `NaN` for every term. `NaN > NaN` is false, so a naive `if (score > best)` loop keeps the *first* candidate by accident (which may coincidentally look right) or keeps `best = -Infinity` and reports "no slot found" → task parked with a coral badge for no physical reason, violating §2.2's parking rule (parking is for *no capacity*, not for a maths artifact). |

### B42 — `balanceScore` on a day whose capacity is zero
| | |
|---|---|
| **Setup** | Cabana → set Sunday window to `start: '10:00', end: '10:00'` (zero-length window; the §8 config exposes windows as tunable). One flexible task "Laundry" (30 min). All of Mon–Sat is 100% full with fixed tasks. |
| **User action** | Add "Laundry" (defaults cascade, §1.1/7A: placed immediately). |
| **Expected** | Sunday's `dayFillRatioAfterPlacement` is `scheduled / capacity` = `30 / 0`. Sunday must be treated as **having no capacity** — not as infinitely balanced. "Laundry" cannot be auto-placed anywhere, so §2.2's failure rule applies: it is parked in the best gap with `schedulingWarning` and a coral flag (FRONTEND §6). It is **never hidden or dropped** ("Visible beats invisible", §0). |
| **Why it's risky** | `30/0 = Infinity`; `1 − Infinity = −Infinity`, so `w.balance · −Infinity` makes Sunday the *worst* day — accidentally correct. But if capacity is computed as `0` and the code guards with `capacity || 1`, Sunday becomes ratio 30 → still negative → fine. The real trap is `0/0 = NaN` when nothing is scheduled yet on Sunday: the day scores `NaN`, comparisons fail silently, and Sunday is either always chosen or never chosen depending on comparison direction. Also: does `getWeekLoad().capacityMin` include a zero-capacity day in `fillRatio`'s denominator? UNSPECIFIED. |

### B43 — Slack for a deadline that has already passed
| | |
|---|---|
| **Setup** | Now = Wed 2026-07-15 14:00. Flexible task "Submit form" (45 min), `deadline = Mon 2026-07-13 09:00` (two days in the past), incomplete, still sitting on the grid. |
| **User action** | Week `⋯` → Re-optimize. |
| **Expected** | §2.4 step 2: `slack = free capacity before deadline − duration`. Free capacity before a past deadline is 0, so `slack = −45` — negative. Urgency test is `deadline ∧ slack < duration × urgencyFactor` → `−45 < 67.5` → **true**, so it sorts into the urgent bucket, ordered slack ASC → it sorts *first* (most negative). §2.2 then finds no pre-deadline slot at all → park in best pre-deadline gap + `schedulingWarning` + coral flag. Crucially, per **P-1** the coral is scheduling physics ("this won't fit"), and the diagnostic must still offer "Let it go" at equal weight. The engine must not throw when the "best pre-deadline gap" set is empty. |
| **Why it's risky** | Three failure modes stack. (1) `Math.max(0, slack)` clamping — common defensive reflex — collapses every past-deadline task to slack 0, destroying the ordering between five different missed deadlines. (2) "Park in the best pre-deadline gap" is *undefined* when there are no pre-deadline gaps in the current week; the likely implementation returns `undefined` and either crashes or silently leaves the task at its old time with no warning badge. (3) §3.6 `carryOver` has a `missedDeadline` flag and says "don't move" — but Re-optimize is a *different* entry point with no such rule, so Re-optimize may yank a missed-deadline task backward in time while carryOver protects it. **The two paths disagree; the spec does not reconcile them. UNSPECIFIED.** |

### B44 — Two byte-identical tasks must sort deterministically
| | |
|---|---|
| **Setup** | Two flexible tasks created via Add Task with *exactly* the same field values: title "Email", 30 min, priority 3, no deadline, no tags, `placedBy:'auto'`. They differ only in `id` (auto: `slug(title) + 4-char suffix` → `email-a1b2`, `email-c3d4`). Tue is empty. |
| **User action** | Re-optimize. Note the resulting order. Re-optimize four more times. |
| **Expected** | §2.4 step 2's sort chain — urgent → priority DESC → deadline ASC → duration DESC → **title ASC** — exhausts itself at "title ASC" with an exact tie. The result must still be **stable and identical on every run**: the same task takes 08:00 both times, forever. Whatever the last-resort tiebreak is (insertion order via a stable sort, or `id` ASC), it must be documented and total. |
| **Why it's risky** | The spec's sort chain has **no total order** — it terminates in a tie for identical tasks. `Array.prototype.sort` is spec-stable in modern V8, so a *pure* comparator chain is fine; but the moment the implementation sorts a derived array, or re-sorts inside the per-placement loop ("day order… recomputed per placement", §2.4 step 3), input order changes and the two tasks swap. Symptom: Re-optimize twice, the grid visibly shuffles, and the user sees the app "changing its mind" for no reason. This also silently breaks any snapshot/planned-vs-actual diff (§7.1). **The spec should append `→ id ASC` to the chain. UNSPECIFIED.** |

### B45 — "Nearest slot" when two candidates are exactly equidistant
| | |
|---|---|
| **Setup** | Wed. Fixed "Standup" 11:00–12:00. Free gaps: 10:00–11:00 and 12:00–13:00 only (everything else on Wed is fixed and full). Origin for scoring = 11:00 (task dragged from / created at 11:00). Flexible task "Write notes" (60 min). Weights: proximity 1.0, all others 0. |
| **User action** | Drop "Write notes" onto Wed 11:00 (onto the fixed Standup) → snap-back per §3.1 → then Re-optimize. |
| **Expected** | Both gaps have `|slot.start − origin|` = 60 min, so `proximity` term is identical. §2.3: **ties → earlier**. "Write notes" takes 10:00–11:00. This must hold even though the *later* slot might feel more natural ("after the thing I dropped on"). |
| **Why it's risky** | "ties → earlier" is unambiguous, but `|slot.start − origin|` measures **start-to-start**, so a 60-min task at 10:00 actually *ends* at the origin — zero distance by any end-aware metric. An implementation that measures "distance from origin to the nearest edge of the slot" (a very reasonable-sounding refinement) makes 10:00 strictly better, which is the same answer for the wrong reason — and then differs from the spec the moment durations are unequal. Worse: floating-point proximity values that "tie" at `0.9999999999999998` vs `1.0` won't compare equal, so the earlier-wins rule silently never fires. Ties must be tested with an epsilon. |

### B46 — Urgency ordering when slack recomputation flips the queue mid-run
| | |
|---|---|
| **Setup** | Mon 08:00–18:00 is the only free day (10h capacity). Three flexible tasks, all `deadline = Mon 18:00`: "A" 240 min, "B" 240 min, "C" 240 min. Total 720 min > 600 min capacity. `urgencyFactor = 1.5`. |
| **User action** | Re-optimize. |
| **Expected** | §2.4 step 2 requires slack to be **"recomputed after each placement"**. Initially each has slack = 600 − 240 = 360; `360 < 240×1.5 = 360`? No — `<` is strict, so **none is urgent** on the first pass and they fall to priority DESC → deadline ASC → duration DESC → title ASC → A, B, C. A places (Mon 08:00–12:00). Now free-before-deadline = 360, so B's slack = 120 < 360 → B becomes urgent. B places (12:00–16:00). C's slack = 120 − 240 = **−120** → no pre-deadline capacity → C parks with `schedulingWarning` + coral flag (§2.2). Exactly one coral flag; A and B are clean. |
| **Why it's risky** | This case sits *exactly* on the `<` boundary (`360 < 360` is false) and then crosses it after one placement — precisely the behavior "recomputed after each placement" exists to produce. The lazy implementation sorts **once** up front, finds nobody urgent, and gets the same visible answer here by luck; swap A's duration to 200 min and the two implementations diverge. Second trap: "free capacity before deadline" — does it mean *total gap minutes* before the deadline, or *contiguous* gap minutes large enough to hold the task? For C the answer is the same (0 either way), but for a fragmented day (six 40-min gaps, 240 min "free", task needs 240 contiguous) total-minutes slack says "relaxed" while reality says "impossible". **The spec does not say contiguous. UNSPECIFIED — and this is the single most load-bearing ambiguity in §2.4.** |

### B47 — Stability bonus applied to a task the engine is forbidden to move
| | |
|---|---|
| **Setup** | Flexible task "Gym" 60 min, dragged by the user to Thu 17:00 → `placedBy` becomes `'user'` (§1.1: "set 'user' on any manual drag/time-edit"). Weights at default (`stability: 0.15`). |
| **User action** | Re-optimize. |
| **Expected** | §2.4 step 2: candidates are "flexible ∧ unpinned" — "Gym" *is* a candidate; `placedBy` is explicitly **"soft"** (§1.1). So the engine may move it, but §2.3's `stabilityBonus(placedBy === 'user')` weights its *current* slot upward so it tends to stay put. **R-1 ("Manual always wins") is about drop/edit conflict outcomes, not about Re-optimize.** Expected: "Gym" stays at Thu 17:00 unless another slot beats it by more than the stability margin — and if it does move, that is correct behavior, not a bug. |
| **Why it's risky** | Two opposite over-corrections, both defensible-looking. (a) The implementation reads R-1 as "never touch user-placed tasks" and excludes them from candidates entirely — now `stability` is dead weight in the scoring formula (it can never differ between candidates, since candidates are by definition auto-placed), and Re-optimize progressively ossifies as users drag things. (b) The implementation applies `stabilityBonus` to the *task being placed* rather than to *the slot it currently occupies*, making it a constant added to every candidate slot — mathematically a no-op after renormalization. Note §2.3 writes `stabilityBonus(placedBy === 'user')` with **no slot argument**, which literally reads as (b). **The formula as written is a constant per task, not per slot. UNSPECIFIED — needs to be `stabilityBonus(task, slot)`.** |

### B48 — Proximity when the candidate slot is beyond the lookahead horizon
| | |
|---|---|
| **Setup** | `config.maxPlacementLookahead = 3` days. Now = Mon 09:00. Mon, Tue, Wed are 100% full with fixed tasks. Thu 08:00–18:00 is completely empty. Flexible task "Plan trip" 60 min, no deadline. |
| **User action** | Add "Plan trip". |
| **Expected** | Per §2.3, `proximity = 1 − |slot.start − origin| / lookaheadHorizon`. Thu 08:00 is ~71 h from origin; the horizon is 72 h, so proximity ≈ 0.01 — small but positive and legal. "Plan trip" places on Thu. If the only free slot were Fri (beyond 72 h), proximity goes **negative** — the term must not be clamped in a way that makes distant slots tie at 0, and the task must still be placed rather than parked, because §2.2 only parks for *deadline* infeasibility, not for horizon overrun. |
| **Why it's risky** | `maxPlacementLookahead` is named like a **hard search bound** (stop looking after 3 days) but used in §2.3 as a **soft scoring scale** (the proximity denominator). If implemented as a hard bound, a task with no deadline and no free slot inside 3 days gets parked with a coral warning — which is a **P-1 violation**: coral is reserved for "this won't fit", and Thursday fits fine. If implemented as a pure scale, negative proximity beyond the horizon can make a far slot score *worse than* a terrible near slot in an overpacked week, quietly stuffing tasks into 5-minute-break Monday instead of using empty Thursday. **The spec uses the value for scoring only; whether it also bounds the search is UNSPECIFIED.** |

## B. Zones (B49–B54)

### B49 — A task whose tags match two overlapping exclusive zones
| | |
|---|---|
| **Setup** | Zone "Study" (`matchTags: ['study']`, window Tue 09:00–13:00, `exclusive: true`, default aqua tint). Zone "Deep Work" (`matchTags: ['focus']`, window Tue 11:00–15:00, `exclusive: true`, custom tint). Flexible task "Thesis draft" (90 min, `tags: ['study','focus']`). |
| **User action** | Add "Thesis draft" → Re-optimize. |
| **Expected** | The task matches both (§1.2: `task.tags ∩ matchTags ≠ ∅`). Both zones are legal hosts, so the union of their windows (Tue 09:00–15:00) is the candidate space and §2.3 scoring picks within it. **No info badge** — the task is inside a matching zone either way, so the "placed outside Study zone" badge (§2.2) must not fire. In the overlap region 11:00–13:00 the two tints must composite legibly at ~12% opacity each (SPEC §10) / as halftone density (FRONTEND §5) and still leave ink text at AA. |
| **Why it's risky** | The engine almost certainly does `zones.find(z => z.matches(task))` — **first match wins** — silently discarding Deep Work and shrinking the legal space to 09:00–13:00. That's invisible in the happy case and only shows up as "why won't it use 14:00, there's nothing there?". Zone *order* then becomes load-bearing state that no UI exposes. Visually: two 12% tints stacked read as ~23% and a hanging-sign label per zone in the same region will collide. **The spec never says whether multi-match unions or first-wins. UNSPECIFIED.** |

### B50 — Exclusive zones for two different tags cover the entire day
| | |
|---|---|
| **Setup** | Zone "Study" (`matchTags: ['study']`, Wed 08:00–13:00, `exclusive: true`). Zone "Admin" (`matchTags: ['admin']`, Wed 13:00–18:00, `exclusive: true`). Together they consume Wed's whole Mon–Fri window. Flexible task "Call plumber" (30 min, `tags: ['errand']`, no deadline). |
| **User action** | Re-optimize. |
| **Expected** | `exclusive: true` means "zone time reserved; non-matching tasks excluded" (§1.2), so Wed has **zero legal capacity** for "Call plumber". It is not urgent and has no deadline, so §2.2's parking rule does not apply — it simply places on Thu (or the best-scoring non-Wed day). No coral flag, no info badge, no error. Wednesday is not "full" in the load bar — `getWeekLoad().perDay[Wed]` should still reflect *scheduled* minutes, not the exclusion. |
| **Why it's risky** | Exclusivity is a per-task filter, but it's tempting to implement it as a mutation of the day's free-slot list — in which case the exclusion for "Call plumber" leaks into the load calculation and Wednesday shows as 100% full when it's empty, poisoning `balance` scoring for **every** task and the load bar for the user. Second trap: if the same task is later given `tags: ['study','admin']`, it matches both zones and the whole day opens up — the free-slot cache must be per-task, not per-day. |

### B51 — A zone with zero windows
| | |
|---|---|
| **Setup** | Cabana → Zones → create zone "Reading" with `matchTags: ['read']`, `exclusive: true`, and **remove every window row** (the shared window-row component, §4.3, allows removal). Flexible task "Journal article" (60 min, `tags: ['read']`). |
| **User action** | Save the zone → Re-optimize. |
| **Expected** | `windowsForDay(day)` returns `[]` for every day. §2.2's precedence is deadline > zone > general windows: a matching zone with no capacity **relaxes** — the task places by general windows with the info badge "placed outside Reading zone" (§2.2 / FRONTEND §6 `--tint-info`). Alternatively the Cabana should refuse to save a windowless zone in the first place. What must **not** happen is the task becoming unplaceable. |
| **Why it's risky** | §2.2's relax rule is written as a *deadline* accommodation ("if a matching zone lacks pre-deadline capacity"). "Journal article" has **no deadline**, so the written rule doesn't cover it — the literal reading is that a matching zone is a hard constraint for non-deadline tasks, and a zero-window zone therefore makes every `read`-tagged task unplaceable forever. Coral flags bloom across the week from a settings action, which is a straight P-1 violation (nothing about this is scheduling physics the user did wrong). Also: does an info badge that says "placed outside Reading zone" make any sense when Reading zone has no hours? It reads as an accusation. **Whether zones relax for non-deadline tasks is UNSPECIFIED and is the highest-value gap in §2.2.** |

### B52 — Zone window shorter than the task it must host
| | |
|---|---|
| **Setup** | Zone "Study" (`matchTags: ['study']`, single window **Thu 14:00–15:00** — 60 min, `exclusive: true`). Flexible task "Mock exam" (**180 min**, `tags: ['study']`, `deadline = Fri 18:00`). Thu and Fri are otherwise empty. |
| **User action** | Add "Mock exam". |
| **Expected** | The zone physically cannot hold a 180-min task. Per §2.2, the zone lacks capacity → **relax** → place by general windows before Fri 18:00 (e.g. Thu 08:00–11:00) with the info badge "placed outside Study zone — due Fri". No coral flag: there is plenty of pre-deadline capacity, so this is not a `schedulingWarning` situation. Info badge (`--tint-info` `#8AA7C2`), not coral (`--tint-warning` `#E2685F`) — the two must never be confused (P-1: coral is physics only). |
| **Why it's risky** | Two wrong answers are easy to reach. (a) **Split the task** to fit the zone — nothing in the spec authorizes splitting a non-project task; chunking is a `chunking` field on projects only (§1.1, §3.7). (b) **Truncate** to 60 min to fit the window — silently destroys 120 min of the user's work, the exact failure that §3.7's conservation rules exist to prevent for projects. (c) Park it with a coral flag: technically "the zone won't fit" is physics, but §2.2 says the zone *relaxes*, and coral for a task that has an obvious legal slot is moral bookkeeping dressed as physics. |

### B53 — Zone tint over a weekend column's halftone screen
| | |
|---|---|
| **Setup** | Zone "Study" (`matchTags: ['study']`, `color` left at default soft aqua) with a window **Sat 10:00–14:00**. Sat's window is 08:00–22:00 (§2.1). Two tasks in that region: one flexible `study` card and one pinned `study` card (frosted glass, R-2: blur 3px + wash `rgba(255,253,247,.55)`). |
| **User action** | View the week at ≥1280 (full week). Then re-check with `prefers-reduced-transparency` set. |
| **Expected** | Per FRONTEND §5, weekend columns get a **light halftone-dot screen (film texture, not color)** and zones render as **denser halftone regions** with a hanging-sign label on hover. So on Saturday two halftone screens stack. The zone region must remain distinguishable from plain weekend — different *density*, not different color — and the hanging-sign label must still be reachable on hover. The pinned card's frosted blur samples whatever is behind it: it must not blur the halftone into mud, and its badges stay on the crisp layer above (R-2), AA-contrast intact. Reduced-transparency fallback must drop the blur without dropping the pinned *meaning* (padlock badge carries it — FRONTEND §9, "icon first, tint second"). |
| **Why it's risky** | SPEC §10 says zones are "tinted background regions (~12% opacity)"; FRONTEND §5 says zones are "denser halftone regions" and weekend is a halftone screen. **These are two different renderings of the same element, and FRONTEND supersedes §10 — but §10's 12% number is the only quantity given, and there's no density number for halftone.** An implementation that keeps the aqua tint (per §10, and per `Zone.color` which still exists in §1.2) puts *color on scenery*, breaking the art direction's one rule: "if it's scenery, it's monochrome." A zone is not clickable and does not warn — so is `Zone.color` dead? **UNSPECIFIED — the `color` field and the hand-tinted rule contradict each other.** |

### B54 — Dragging a non-matching task into an exclusive zone
| | |
|---|---|
| **Setup** | Zone "Study" (`matchTags: ['study']`, Tue 09:00–13:00, `exclusive: true`), empty. Flexible task "Nap" (`tags: ['rest']` — note `rest` ∈ `config.protectedTags`) currently on Tue 15:00. |
| **User action** | Drag "Nap" by its body and drop it at Tue 10:00 — inside the exclusive Study zone. |
| **Expected** | **The drop succeeds.** §2.1: "Windows bound **automatic** placement only — the grid is 24 h and users may drag anywhere." R-1: "A user drop/edit overrides the algorithm unconditionally." Exclusivity governs `autoSchedule()`, not the user's hands. "Nap" lands at Tue 10:00, `placedBy` → `'user'`. §3.1's snap-back list is *targets* that are fixed/pinned/protected — an empty zone is not a task and cannot be a drop target. No warning, no toast, no "are you sure". A later Re-optimize also must not evict it: `rest` is a protected tag → "not post-placement" movable (§1.1 semantics table, row 4). |
| **Why it's risky** | "Exclusive" is a strong word and the natural implementation validates drops against the same free-slot function `autoSchedule` uses — producing a snap-back + shake + toast for a drop the spec explicitly permits. That's R-1 inverted: the algorithm winning over the user. The inverse bug is equally likely: the drop succeeds, then the next Re-optimize quietly evicts "Nap" out of the zone because the free-slot walker still thinks that time is reserved — user work silently undone, and the protected-tag anchor rule (§2.4 step 1) missed. |

## C. Projects & chunking (B55–B60)

### B55 — A total that divides evenly into no legal chunk count
| | |
|---|---|
| **Setup** | Add Project: "Paint the shed", `totalMinutes: 100`, `minChunk: 45`, `maxChunk: 45` (min = max), `range: { from: Mon, until: Fri }`. Week is empty. |
| **User action** | Submit the Add Project panel. |
| **Expected** | 100 is not a multiple of 45. Two chunks = 90 (10 min short); three = 135 (35 min over). §3.7 says children are "sized in [minChunk, maxChunk]" — a closed interval that here contains exactly one value. Something has to give, and the spec's stated bias is **conservation of the user's stated work**: it names "merges residue" and "spawns chunk (≥minChunk)" as the levers. Expected: either (a) the panel refuses at input time with a plain explanation and a suggested total, or (b) it materializes 2×45 + flags the parent `schedulingWarning` ("none fits → parent `schedulingWarning`", §3.7) so the missing 10 min is **visible**. What must never happen: silently scheduling 90 or 135 minutes and calling it 100. |
| **Why it's risky** | §3.7's conservation rules are all written for *edits to existing chunks* (shrink/grow/delete). **Initial materialization has no stated arithmetic at all** — `addProject()` is described only as "materializes children, sized in [minChunk, maxChunk]". The obvious `Math.ceil(total/maxChunk)` gives 3 chunks and then either 45/45/10 (violating `minChunk`) or 34/33/33 (violating `minChunk` differently) or 45/45/45 = 135 (inventing 35 minutes of the user's life). The chunk-preview UI (FRONTEND §6: "buckets filling toward the castle") will happily draw whichever lie it's given. **The materialization algorithm is UNSPECIFIED.** |

### B56 — A range shorter than the work it must contain
| | |
|---|---|
| **Setup** | Add Project: "Grade papers", `totalMinutes: 600` (10 h), `minChunk: 60`, `maxChunk: 120`, `range: { from: Sat 2026-07-18, until: Sun 2026-07-19 }`. Sat window = 08:00–22:00 (14 h) but Sat is already 100% full with fixed tasks. Sun window = 10:00–14:00, `lightDay: true`, `maxTasks: 2`. |
| **User action** | Submit. |
| **Expected** | Sunday offers 4 h of window and, being a light day, `maxTasks: 2` — so at most ~2 chunks (≤240 min) fit, and §2.1 says Sunday is used "only when other days ≥ ~80% full" (it is). The remaining ~360+ min have nowhere legal inside `range.until`. §3.7 says `range.until` "enters slack math", which routes this through §2.2: no capacity before the range end → **park the unplaceable chunks with `schedulingWarning` + coral flag**, and per §3.7's "none fits → parent `schedulingWarning`" the sandcastle parent glyph carries a warning too. Every minute the user asked for is still visible somewhere (§0: "never hidden, never silently dropped"). Per P-1 the diagnostic offers "Let it go" at equal weight to any fix. |
| **Why it's risky** | `range.until` is described as feeding "slack math" — i.e. it acts like a deadline — but a project has **no `deadline` field**; the deadline lives implicitly in `chunking.range.until`. So §2.4's urgency test (`deadline ∧ slack < duration × urgencyFactor`) reads `task.deadline`, which is `null` on materialized chunks unless `addProject` copies `range.until` into each child's `deadline`. If it doesn't, the chunks are non-urgent, sort last, and get scattered into next week past their own range with no warning at all — a completely silent failure. Second trap: `maxTasks: 2` is a *count* cap on Sunday and chunks are numerous and small; a chunker that respects window minutes but not `maxTasks` will stack six buckets on Sunday. |

### B57 — minChunk greater than maxChunk
| | |
|---|---|
| **Setup** | Add Project panel: `totalMinutes: 240`, `minChunk: 120`, `maxChunk: 30`. (User typed them into the wrong fields — an extremely common slip.) |
| **User action** | Submit. |
| **Expected** | `[minChunk, maxChunk]` is the empty interval; no legal chunk size exists. §1.1's precedent for exactly this shape is the `endTime` guard: **"end<=start → swap"**. Consistency says the panel swaps to `min: 30, max: 120`, tells the user plainly, and materializes 2×120. Alternatively, block submit with an inline message. Either is fine; both are far better than materializing. |
| **Why it's risky** | The spec guards `endTime` and clamps `priority` (1–5) but states **no guard on `chunking`** at all. An unguarded chunker loops `while (remaining > 0) { size = clamp(remaining, minChunk, maxChunk) }` — and `clamp(240, 120, 30)` returns 30 in a `Math.min(max, Math.max(min, v))` implementation but **120** in a `Math.max(min, Math.min(max, v))` implementation. One of those makes 8 chunks; the other makes 2. Neither errors. A `while (remaining >= minChunk)` guard with a 0-length interval can spin forever if `size` computes to 0 — a hung tab from a typo in a settings field. |

### B58 — Editing a parent whose first chunk is already completed
| | |
|---|---|
| **Setup** | Project "Build slides", `totalMinutes: 300`, `minChunk: 60`, `maxChunk: 90`, materialized as 4 chunks: C1 90 min (Mon, **completion: 'done'**, rated 4 shells), C2 90 min (Tue, `placedBy:'auto'`), C3 60 min (Wed, dragged by the user → `placedBy:'user'`), C4 60 min (Thu, auto). |
| **User action** | Open the parent in the detail modal and change `totalMinutes` from 300 → 180. |
| **Expected** | 120 min must leave the project. §3.7: **"Completed & user-placed chunks never auto-flexed."** So C1 (done, 90) and C3 (user, 60) are untouchable = 150 min already committed — which **exceeds the new total of 180 by nothing, but leaves only 30 min of headroom**. ## D. The learning module (B61–B65)

### B61 — Cold start crossed by exactly the 10th rating
| | |
|---|---|
| **Setup** | `config.coldStartRatings = 10`. The user has rated exactly **9** completed tasks. `w.preference` is forced to 0 (§5), so §2.3's fourth term contributes nothing and `modelScore` is never consulted. Cabana → Insights currently shows the cold-start state. |
| **User action** | Rate a 10th task (4 shells, `timingFit: +1`) via the shell row, then Re-optimize. |
| **Expected** | §5: `modelScore` "forced 0 until ≥10 ratings" — `≥10` means the 10th rating **crosses** the threshold. But §5 also says retraining happens at **week rollover + Cabana "Retrain now"** only. So immediately after the 10th rating the model has 10 samples but **untrained weights**. Correct: `w.preference` stays effectively inert until a train has actually run — a freshly-initialized ridge model (weights all 0) returns a constant `modelScore`, which after renormalization shifts every slot equally and changes nothing. Cabana should now offer "Retrain now (10 samples)" and the plain-language preferences panel should populate only after training. |
| **Why it's risky** | The gate is on **rating count**, the model quality is gated on **training runs**, and the two are only coupled by a weekly timer. The bad implementation flips `w.preference` to 0.15 the instant count hits 10 and calls `modelScore` on an untrained model — which may return `NaN` (untrained weights `undefined`), garbage outside `[0,1]` (§5 promises `∈[0,1]`), or 0.5 for everything. Because `w.preference` is renormalized *into* the other three weights, a 0.15 slice is stolen from proximity/balance/stability to be spent on noise: the user rates one task and their whole week reshuffles for no reason they can perceive. That is the app appearing to "change its mind", the thing P-1's whole tone is trying to avoid. |

### B62 — Ten ratings that all say exactly the same thing
| | |
|---|---|
| **Setup** | The user rates their first 10 completed tasks **5 shells each**, with all three facets left untapped (facets are "optional always", §5). Every target `overall` maps to 1.0. The tasks vary in tag, time-of-day bucket, day, duration, and priority. |
| **User action** | Cabana → **Retrain now**. |
| **Expected** | Target variance is **zero**. Ridge regression on a constant target converges to: **all feature weights ≈ 0, intercept ≈ 1.0**. `modelScore(task, slot)` returns ~1.0 for every task and every slot — a constant. Since §2.3 adds `w.preference · modelScore` to every candidate slot equally, the term **cannot change any ranking**: placement is identical to the cold-start behavior. That is the mathematically correct and desirable answer. The Cabana's plain-language panel must say something honest — "You've liked everything so far; no preferences learned yet" — and **not** invent a preference. Sample count shows 10. |
| **Why it's risky** | Two failure shapes. (1) **Feature normalization divides by zero variance** — z-scoring features or targets on a constant column gives `NaN`, which propagates through gradient descent and poisons *every* weight, not just the constant one; `modelScore` then returns `NaN` and slot scoring degenerates (see B41). (2) **The plain-language renderer** (§5: "weights inspectable → Cabana renders plain-language preferences") reads near-zero weights and rounds noise into confident prose: "You prefer Tuesday afternoons" from a weight of 0.00004. That's the app *lying* about the user to the user, in a panel whose entire justification is inspectability. Ridge's L2 penalty helps here — it shrinks toward zero — but only if the intercept is **excluded from regularization**, which naive implementations forget; a regularized intercept can't reach 1.0 and every weight fights to make up the gap. |

### B63 — Ratings that flatly contradict each other
| | |
|---|---|
| **Setup** | Twelve rated completions of tasks tagged `gym`, all at the 18:00–21:00 time bucket, all 60 min. Six are rated **5 shells, `timingFit: +1`**. Six are rated **1 shell, `timingFit: −1`**. Nothing else distinguishes them. §5: "`timingFit≠0` **doubles sample weight** on time features" — so all twelve carry double weight on the time-of-day feature. |
| **User action** | Cabana → Retrain now, then read the plain-language preferences. |
| **Expected** | The evening-gym feature is perfectly ambiguous: the least-squares fit puts its weight at ~0 and the intercept at ~0.5. `modelScore` for an evening gym slot ≈ 0.5 — genuinely uninformative, which is the truth. The Cabana must render this as **no stated preference** for evening gym (or explicitly "mixed"), never as a weak preference in whichever direction the last gradient step happened to land. Doubling the weight of *both* sides changes nothing about the mean — it must not amplify one side. |
| **Why it's risky** | "Doubles sample weight" is easy to implement as *duplicating rows*, which is equivalent here — but if implemented as doubling the **learning rate contribution** rather than the sample weight, gradient descent on contradictory samples **oscillates** and the final weight depends on which sample was last in the batch and on the epoch count. Retrain twice, get opposite prose: "You prefer evening workouts" / "You avoid evening workouts". A user who reads both statements loses all trust in the module, and §5's whole design premise is that weights are **inspectable** — inspectable nonsense is worse than a black box. Also: with `overall ∈ [0,1]` from a 1–5 shell scale, is 1 shell → 0.0 or → 0.2? **The shell→target mapping is UNSPECIFIED**, and it decides whether the intercept lands at 0.5 or 0.6. |

### B64 — A rated task is deleted
| | |
|---|---|
| **Setup** | 11 ratings exist (past cold start). One of them belongs to task "Deep clean kitchen" (completed Mon, rated 2 shells, `energy: −1`). The model has been trained on all 11. |
| **User action** | Right-click "Deep clean kitchen" → Delete. Then Cabana → Retrain now, and check the sample count. |

| **Why it's risky** | The alternative design — training samples are **immutable events**, decoupled from the task rows — is defensible and arguably better (you learned something real; deleting the card doesn't unlearn it). If samples live inside `task.satisfaction` (§1.1 — they do; that's where the field is), deletion is coupled by construction, the count silently drops, and a user who tidies up their week can **fall back below cold start and lose their trained model with no warning at all** — a far bigger loss than §3.10 bothers to confirm. If samples were copied out to a training store, the sample count in "Retrain now (N samples)" will disagree with the number of rated cards on the grid, which is its own trust bug. **UNSPECIFIED, and the two answers have opposite storage designs — this must be decided before §5 is built.** |

### B65 — What an inspectable weight actually says to a human
| | |
|---|---|
| **Setup** | Trained model, 40 samples. Feature weights include: `tag:study = +0.31`, `timeBucket:early-morning = −0.22`, `dayFillAtCompletion = −0.55`, `moveCount = −0.08`, `placedBy:user = +0.04`. Cabana → Insights → preferences panel. |
| **User action** | Read the panel. |
| **Expected** | §5: "weights inspectable → Cabana renders plain-language preferences." | **Why it's risky** | Raw ridge weights are only comparable if features are on the same scale — and these are not: `dayFillAtCompletion ∈ [0,1]`, `moveCount ∈ [0, ∞)`, `tag:study ∈ {0,1}`. The system inverts but must remain **coherent and non-crashing**; the honest fix is for the Cabana to clamp or refuse the inversion (`minimum ≤ medium ≤ default`) exactly as §1.1 guards `endTime` and clamps `priority`. |
| **Why it's risky** | §3.3 says "**compress** ... to `breaks.minimum`" — the word "compress" implies `Math.min(currentBreak, minimum)`, which with minimum=45 **cannot expand** and so silently no-ops, while §2.4's threshold ladder *does* read the value directly and demands 45. The two subsystems then disagree about what the padding is on the same day, and the ripple's cost model (§3.2: `cost(ripple)` = downstream shift minutes + evacuations × penalty) is computed against padding that stage (1) won't actually deliver → the chooser pre-highlights the wrong default (§3.2 says the cheaper option is pre-highlighted), and the user's Enter key does the more expensive thing. §7.3's overpack detector also reads `minimum × 1.5` = 67.5 min → **every** day now looks overpacked → the whistle notice fires on an empty week, which is a P-1 violation manufactured by a slider. No guard is specified. |

### B74 — Footlocker import replaces config with a config from a different week's tuning
| | |
|---|---|
| **Setup** | Current state: 12 tasks, 2 zones, `weights.proximity: 0.5` (default), 11 satisfaction ratings (model trained, `w.preference` live). Export file `schedule-2026-07-01.json` from three weeks ago: 8 tasks, 0 zones, `weights.proximity: 0.9`, `urgencyFactor: 3.0`, 4 ratings, `schemaVersion: 1`. |
| **User action** | Cabana → treasure chest → **Key** (import) → choose the file → read the dry-run summary → confirm **Replace**. |
| **Expected** | §9: "import = validate → **dry-run summary** → replace/merge". The dry-run must state what actually changes, and `config` is part of the state (§1.3: `Schedule { tasks, zones, config }`) — so the summary must disclose the **config and tuning changes**, not only the task/zone counts. Concretely it must warn that ratings drop 11 → 4, which crosses `coldStartRatings: 10` **downward** and silently switches `w.preference` back to a forced 0 (§5) — the trained model is gone. It must also disclose `urgencyFactor: 1.5 → 3.0`. After Replace: the model is cold-start again, and the Cabana's plain-language preferences panel must empty rather than keep showing stale prose. |
| **Why it's risky** | "Dry-run summary" is universally implemented as a **task diff** ("8 tasks will replace 12") because tasks are what the user sees. Config and model state ride along invisibly: the user imports a backup to recover a deleted task and, without a word, their weights, their zones (12 → 0!), and their trained model are replaced by three-week-old values. Worse, `weights.proximity: 0.9` and `urgencyFactor: 3.0` don't announce themselves — the week just starts behaving differently and the user has no idea a slider moved. **Whether `config` is in scope for import/export at all is UNSPECIFIED** — §9 says "full state" for auto-save but never enumerates the export payload. If it is, the dry-run needs a config section; if it isn't, the export isn't a backup. |

## G. Responsive & mobile (B75–B77)

### B75 — Single-day layout below 768 must still show cross-day consequences
| | |
|---|---|
| **Setup** | Viewport 390×844 (<768 → single-day layout with day-picker strip, FRONTEND §5; film sprockets collapse to top-only). Viewing **Wednesday**. Wed 14:00 holds flexible "Report draft" (90 min). Thursday is 95% full. |
| **User action** | Resize "Report draft" from 90 → 240 min via the sand strip (bottom border, 8px, ns-resize, 15-min snap — on touch, an 8px target). §3.2's encoded case says a **+4h resize → displace**; here +2.5h with a full Thursday means the chooser appears and either branch pushes work **onto Thursday**, a day not on screen. |

On a 390px viewport it must remain fully legible and must **name the off-screen consequence** — "3 tasks move to Thursday" — because the user cannot see Thursday. After commit, the displaced/relocated cards get the 180ms squash-and-stretch settle (FRONTEND §8) that the user **will not see** on another day, so the feedback burden falls entirely on the toast/plaque. |
| **Why it's risky** | Every conflict affordance in the spec is written for a visible week: "the cheaper option is pre-highlighted", cards "animate with a brief squash-and-stretch settle" — both assume you can *watch* the consequence. On the primary mobile layout (FRONTEND §5 calls <768 "primary mobile layout"), the entire result of a resize can happen off-screen with a 180ms animation nobody sees, and the user's only evidence is a chooser they Enter-ed through. Second, harder problem: an 8px resize strip is roughly a quarter of the ~44px minimum touch target, and it sits **inside** the card whose body-drag means *move* — on touch, "resize" and "move" become the same gesture with a 4px difference. **The spec gives no touch-specific resize affordance. UNSPECIFIED**, and it makes the primary layout's core interaction a coin flip. |

### B76 — Weekend drawer at 768–1279 while a zone and an overpack notice both want the weekend
| | |
|---|---|
| **Setup** | Viewport 1024×768 → **Mon–Fri + weekend drawer** (drawer pull drawn as a beach-towel tab, FRONTEND §5). Zone "Study" has a **Sat 10:00–14:00** window. Sat holds 5 tasks with average break 6 min; Sun holds 2 (its `maxTasks` cap). §7.3's overpack notice has fired citing 3 days, **two of which are Sat and Sun** — inside the closed drawer. |

| **Expected** | The notice is "the one grid-side notice (physics)" (§7.3) and must be honest about days the user cannot currently see — naming Sat/Sun explicitly, and ideally opening the drawer when tapped. Zone regions that only reveal their identity on hover are also a straight FRONTEND §9 problem at this breakpoint ("meaning = icon first"), since a denser halftone with no reachable label carries no meaning at all. **Whether drawer-hidden days count for §7.3 is UNSPECIFIED.** |

### B77 — A drag that starts at one breakpoint and ends at another
| | |
|---|---|
| **Setup** | Viewport 1300px wide → **full week** layout. The user picks up flexible "Groceries" (Fri 16:00, 60 min) — either by mouse-down with the ghost active (0.85 opacity, 1.03 scale, raw follow, FRONTEND §8) or by keyboard (Space to pick up, §10 accessibility). Mid-drag, the browser window is resized to **1200px** — crossing into Mon–Fri + weekend drawer — or a tablet is rotated, crossing 768. |
| **User action** | Complete the drop (mouse-up over what *was* Saturday's column / press Enter). |
| **Expected** | The drag is either **cancelled cleanly** — snap-back to Fri 16:00, ghost removed, `placedBy` unchanged, no toast, per Esc semantics (§3.2/§10: "Esc cancel") — or it **completes against the new layout's real coordinates**. What must not happen: a drop resolved against **stale column geometry**, silently writing "Groceries" to a day the user never pointed at, with `placedBy: 'user'` (§1.1) making that phantom placement **sticky** — the stability bonus (§2.3) then protects the error from every future Re-optimize, and R-1 says manual wins, so the engine will defend it forever. |
| **Why it's risky** | Drag state is held in JS while column geometry is held in CSS; a breakpoint change swaps the CSS and nothing tells the drag. ## H. Multi-week & lifecycle (B78–B80)

### B78 — `carryOver` run twice for the same pair of weeks
| | |
|---|---|
| **Setup** | Week 29 (past) has 3 incomplete flexibles: "Fix bike" (60 min), "Call bank" (30 min), "Read paper" (45 min, `deadline = Fri 2026-07-17 17:00` — **passed**). Each has `history.carriedCount = 0`. Week 30 is current. |
| **User action** | Week `⋯` → **Wrap up week** (runs `carryOver(W29, W30)`). Then, without reloading, do it again: navigate back to W29 and run **Wrap up week** a second time. |
| **Expected** | §3.6 is precise per class: recurring → regenerate naturally; **deadline-passed → flag `missedDeadline`, don't move**; rest → re-place as `placedBy:'auto'`, incrementing `history.carriedCount`. First run: "Fix bike" and "Call bank" move to W30 (`carriedCount` 0→1); "Read paper" stays in W29 flagged `missedDeadline`. Second run must be **idempotent-in-effect**: W29 no longer contains "Fix bike"/"Call bank" (they left), so there is nothing to carry — `carriedCount` stays at **1**, not 2. "Read paper" is still in W29 and still deadline-passed → still doesn't move, and `missedDeadline` is re-set to the same value (harmless). No duplicate tasks appear in W30. |
| **Why it's risky** | `carryOver(fromWeek, toWeek)` takes explicit arguments and the `⋯` menu is always available, so **running it twice is a two-click user action, not an edge case** — and the past-week banner (§3.6, "once, dismissible") is a *second* entry point to the same operation, so banner + menu is the likely double-fire. The failure isn't duplication (the task left W29, so a correct implementation finds nothing); it's **`carriedCount` inflation** if the counter is incremented on the *scan* rather than on the *move*, or if carryOver is implemented as "for each incomplete task in the source **week range**" while a carried task's new home in W30 is *also* in scope of a sloppy range query. `carriedCount ≥ 3` (combined with `displacedCount`) trips the **starvation detector** (§7.2) → the user gets "Fix bike is being starved — Pin next week / Let it go" about a task they carried once. That's a false accusation generated by a double-click, in an app whose first principle is that it never guilts. |

### B79 — Week rollover at exactly midnight Sunday→Monday with the tab open
| | |
|---|---|
| **Setup** | The tab is open, showing week 29, at **Sun 2026-07-19 23:59:30**. The user is mid-edit: the detail modal is open on flexible "Meal prep" (Sun 21:00) with an unsaved title change typed in. Auto-save is debounced 2 s (§9). 12 ratings exist. |
| **User action** | Wait 30 seconds. The clock crosses **Mon 2026-07-20 00:00:00**. |
| **Expected** | §2.4: autoSchedule "runs on: initial load, explicit Re-optimize, **week rollover**". §5: retraining runs at "week rollover (before Wrap report)". §7.1: the Wrap report is "generated at rollover (post-retrain)". So at 00:00 a chain fires: **retrain → Wrap report → autoSchedule**, plus (per §3.6's triggers) the past-week banner becomes eligible. All of that must **not** destroy the open modal or the unsaved edit, must not yank the view from W29 to W30 under the user's cursor, and must not throw a generated PDF at someone at midnight. P-1 says insights live in the report and the Cabana, "never as grid nagging" — and a PDF appearing unbidden at 00:00 is the loudest possible nag. Boundary precision matters: `23:59:59.999` is week 29 and `00:00:00.000` is week 30, with **no gap and no overlap** (§1: "All date math local-time, minute precision, seconds zeroed" — note that rule makes 23:59:30 and 23:59:00 the *same minute*, so a minute-precision clock never observes :30). |
| **Why it's risky** | Rollover is specified as a moment but the app is a **stateful UI with an open modal and a debounced write**. The autoSchedule leg is the dangerous one: it re-places every flexible ∧ unpinned task, and if "Meal prep" is among them, the row under the open modal is mutated while the user is typing into it — the 2s-debounced save then writes either the pre-rollover object (resurrecting the old time) or a merge of both (§1.1's `clone()` exists for "optimistic edits" and is explicitly "**never enters tasks[]**" — a rollover racing an optimistic edit is exactly how a clone leaks into `tasks[]` with a **duplicate id**, since `clone()` keeps the same id). Second: DST. `config.windows` are wall-clock strings ('08:00') and all date math is local — a rollover on a DST-shift weekend makes a 23-hour or 25-hour Sunday, so "midnight" is either skipped or occurs twice. **The spec says local-time and never mentions DST. UNSPECIFIED.** |

### B80 — A task carried into a week that is currently being edited
| | |
|---|---|
| **Setup** | Two tabs of the PWA are open on the same browser profile (§12: installable PWA; §9: localStorage primary — **shared across tabs**). Tab A shows week 30 and the user is dragging "Groceries" to Wed 15:00. Tab B shows week 29 and the user clicks **Wrap up week** → `carryOver(W29, W30)` re-places "Fix bike" (60 min) via scoring, and it lands at **Wed 15:00** — the slot Tab A's drag is targeting. |
| **User action** | Complete the drop in Tab A. |
| **Expected** | Both writes go through §9's auto-save (debounced 2 s) to the same localStorage key. The **last write wins** and one user action is silently lost — so either (a) the two tabs share state via a `storage` event and Tab A sees "Fix bike" arrive mid-drag (then §3.1's normal drop-conflict resolution applies: target is flexible ∧ unpinned → evict via scored placement — a legitimate outcome), or (b) the app detects the concurrent session and refuses/warns. What must not happen: Tab A's whole `tasks[]` array — a stale snapshot taken before "Fix bike" existed — is written wholesale, **deleting "Fix bike" from existence** with no error and no trace. `history.carriedCount` was already incremented in the losing write, so even a re-carry would double-count (see B78). |
| **Why it's risky** | §9 specifies a StorageAdapter, a 2s debounce, and a **status dot (green persistent / amber session)** — a UI that reports *whether* state persists but never *whose* state. "Auto-save **full state**" is the killer phrase: a full-state write from a stale in-memory `Schedule` is a **blind overwrite**, and localStorage offers no compare-and-swap. Two tabs is not exotic — it's a PWA with an installed window plus a browser tab, or a laptop left open while the user opens the app on the same machine's second profile. The spec never mentions multi-tab, the `storage` event, or write conflicts, and the entire persistence design (single key, full-state, debounced, last-write-wins) is the textbook shape for silent data loss. **UNSPECIFIED — and unlike most gaps here, the failure mode is destroyed user data with zero signal.** |

## Top 10 most likely to be broken

1. **B41 — all weights zero → `NaN` scoring.** `Σw = 0` in a renormalization step is a one-line division the spec mandates and nobody guards; every slot scores `NaN`, comparisons silently fail, and the whole engine degenerates from a Cabana slider the spec explicitly says is tunable.
2. **B44 — identical tasks sort nondeterministically.** The §2.4 sort chain terminates in a tie with no `id` tiebreak, and step 3 re-sorts per placement — so the grid visibly reshuffles between two Re-optimizes and every snapshot diff is noise.
3. **B60 — conservation silently drops 60 minutes.** Three levers blocked by three different rules (maxChunk ceiling, user-placed immunity, minChunk floor); getting the order wrong makes the user's finals prep quietly shrink, and the grid looks *tidier*, not wronger.
4. **B80 — two tabs, full-state blind overwrite.** Single localStorage key + debounced full-state write + no compare-and-swap is the textbook shape for destroyed data, and the spec never once mentions multi-tab despite shipping as an installable PWA.
5. **B69 — missed rollovers.** Every weekly behavior (retrain, report, autoSchedule, carryOver) hangs off an event that only fires while a tab is open; the two available implementations are "never runs" and "runs three times", and the spec picks neither.
6. **B51 — zero-window zone makes matching tasks unplaceable.** §2.2's relax rule is written only for the deadline case, so the literal reading turns a settings action into coral flags across the week — a P-1 violation manufactured by the Cabana.
7. **B46 — slack recomputation and the contiguity question.** "Free capacity before deadline" is unspecified as to contiguity; total-minutes slack says a fragmented day is fine while physics says it's impossible — and the strict `<` boundary hides the difference in exactly the test case one would write.
8. **B71 — `urgencyFactor: 0` disables deadline enforcement.** The slider is named for urgency but the natural implementation gates the whole deadline branch on the urgency predicate, so 0 silently turns hard constraints into nothing with no warning.
9. **B43 — past deadlines: clamped slack + Re-optimize vs carryOver disagreement.** `Math.max(0, slack)` is the reflexive guard and it collapses every missed deadline into one indistinguishable bucket; meanwhile §3.6 protects missed-deadline tasks and §2.4 doesn't, so the two entry points fight.
10. **B62/B63 — degenerate ratings produce confident false prose.** Zero-variance targets give `NaN` through feature standardization; contradictory ratings oscillate; either way the Cabana's inspectability panel states a preference the model doesn't hold — the app lying about the user in the one place whose entire purpose is honesty.
